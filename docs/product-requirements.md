# 产品需求说明

## 项目目标

构建一套基于 Rust 的 trade[XYZ] / Hyperliquid 自动交易系统。系统第一阶段支持：

1. 斐波那契回撤接针策略。
2. 聪明钱地址跟单策略。
3. 前端控制台，用于账号总览、基础手动交易、斐波那契策略管理、聪明钱跟单管理。

系统必须具备可扩展策略架构，后续可以继续加入更多策略模块，而不重写基础信息获取、
交易执行、风控、状态存储等核心基础设施。

## 功能 1：斐波那契回撤接针

### 需求

用户可按不同时间维度配置斐波那契回撤策略。

策略支持：

- 多个时间维度，例如 1m、5m、15m、1h、4h、1d。
- 回撤档位：默认 0.382、0.618，可扩展到 0.236、0.5、0.786。
- 当价格到达回撤点附近时自动执行买入。
- 用户可配置“附近”的容忍范围。
- 买入实际成交后，以实际成交价为基准计算卖出条件。
- 当实际成交价上涨 N 美元后自动卖出止盈。
- 当实际成交价亏损 X% 后自动止损。

### 关键约束

- 入场、止盈、止损都必须通过统一风控网关。
- 止盈止损必须基于成交价，不基于信号价。
- 同一回撤档位是否允许重复买入由策略配置控制。
- 未拿到实时 metadata、行情、账户状态时不得新开仓。

## 功能 2：聪明钱跟单

### 需求

用户可配置一批目标跟单地址，系统监听这些地址的交易行为，并按规则复制买入、卖出、
减仓、平仓行为。

策略支持：

- 聪明钱地址配置。
- 地址启用/禁用。
- 目标地址行为识别。
- 严格执行目标地址买入、卖出策略。
- 每个交易对最大跟单数量。
- 多账号跟单去重。
- 账号跟单比例。
- leader 分组去重。
- 交易对维度、leader 维度、账户维度风控。

### 关键约束

- 跟单策略只发出意图，不直接下单。
- 重复 leader 事件不得导致重复下单。
- 平仓和减仓必须使用 reduce-only。
- leader 行为无法可靠识别时必须保守拒绝。
- leader 事件延迟超过阈值时不得追单。

## 功能 3：前端控制台

### 需求

系统提供一个本地前端控制台，用于展示账号基础情况、人工点击买入/卖出/撤单、设置
止盈止损、管理斐波那契策略、管理聪明钱跟单策略，并支持多账号手动操作。

控制台支持：

- 查看当前环境、dry-run/live 状态。
- 查看多账号资金、账户权益、保证金状态、仓位、PnL、open orders。
- 查看启用交易对的行情、mark、mid、oracle、spread。
- 点击买入。
- 点击卖出。
- 点击撤单。
- 批量多账号操作。
- 为已有仓位设置止盈止损。
- 配置、启停和监控斐波那契策略。
- 配置、启停和监控聪明钱跟单策略。
- 测试基础信息模块和基础交易模块是否可用。

### 关键约束

- 前端不得直接调用交易所下单 API。
- 人工买卖必须生成 `ManualTradeIntent` 并进入统一风控网关。
- 策略页面只能提交配置变更、启停命令或 kill switch 命令，不得直接下单。
- 多账号批量操作必须由 coordinator fan-out 到每个 account worker，逐地址风控、
  逐地址执行、逐地址审计。
- 主网 live 人工操作必须二次确认。
- 前端不得接触或保存私钥。

## 模块需求

### 模块 1：基础信息获取模块

包含：

- 实时行情获取。
- K 线或时间维度聚合。
- 市场 metadata 获取。
- 目标账户交易行为监听。
- 本账户状态获取。
- REST 对账。

输出：

- `MarketEvent`
- `LeaderEvent`
- `AccountEvent`

### 模块 2：基础交易模块

包含：

- 买单。
- 卖单。
- 撤单。
- 改单。
- maker / taker / IOC / GTC / ALO。
- reduce-only。
- nonce。
- `cloid`。
- 订单回报。

输入：

- `ApprovedOrder`
- `CancelCommand`

### 模块 3：斐波那契回撤策略模块

包含：

- 回撤点计算。
- 接针买入信号。
- 止盈点计算。
- 止损点计算。
- 策略状态维护。
- 策略专属风控。

输出：

- `TradeIntent`

### 模块 4：聪明钱跟单模块

包含：

- 聪明钱地址配置。
- 聪明钱行为识别。
- 跟单比例计算。
- 多账号去重。
- 交易对限制。
- 策略专属风控。
- 买入/卖出/减仓/平仓意图。

输出：

- `TradeIntent`

### 模块 5：前端控制台与基本操作模块

包含：

- Dashboard：首页账号总览。
- 本地 Web 前端。
- 多账号行情、余额、仓位、订单展示。
- PnL 展示。
- 点击买入。
- 点击卖出。
- 撤单。
- 批量多账号操作。
- 手动设置止盈止损。
- 斐波那契策略配置、启停、监控页面。
- 聪明钱跟单配置、启停、监控页面。
- 手动 dry-run / testnet 测试。

输出：

- `ManualTradeIntent`
- `CancelCommand`
- `StrategyConfigChange`
- `StrategyControlCommand`

关键要求：

- 不得直接调用基础交易模块下单。
- 人工买卖必须通过统一风控网关。
- 策略配置变更必须校验并写审计日志。
- 多账号批量操作必须 fan-out 到每个账号对应的 account worker。
- 主网 live 人工操作必须有二次确认和完整审计日志。

### 横切模块

为保证可扩展性，必须新增以下横切能力：

- 事件总线。
- 状态存储。
- 统一风控网关。
- 配置系统。
- 前端控制台 API 和本地前端。
- 日志和审计。
- 测试和回放框架。

## 架构验收标准

基础信息模块 + 基础交易模块 + 风控网关 + 状态存储，应能形成最小交易闭环。

策略模块必须是可插拔的：

- 添加新策略不需要修改基础交易模块。
- 添加新策略不需要修改基础信息模块。
- 添加新策略只需要注册新的 `Strategy` 和 `StrategyRiskCheck`。

前端控制台必须复用同一条交易链路：

- 手动操作不需要新增交易所执行路径。
- 手动操作不需要绕过风控。
- 手动操作可以作为基础信息模块和基础交易模块的测试入口。
- 手动多账号操作必须由多个 account workers 并行执行，不允许单进程串行替多个
  地址下单。
- 策略页面只修改策略配置和状态，不绕过策略引擎直接下单。

## 非目标

第一阶段不追求：

- 复杂图形化交易终端。
- 复杂机器学习策略。
- 全自动主网大资金交易。
- 跨交易所套利。

第一阶段重点是：

- 架构稳定。
- 数据和执行闭环可靠。
- 风控清晰。
- 可回放、可对账、可恢复。
